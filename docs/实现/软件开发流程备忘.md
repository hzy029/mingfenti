# 软件开发流程备忘（单人 / AI 辅助）

设计阶段之后「该做什么」可以用下面这条主链理解；不要求每步都写正式文档，但**顺序**能减少返工。

## 1. 常见全流程（从小到大）

| 阶段 | 做什么 | 产出物（可简可繁） |
|------|--------|---------------------|
| **立项 / 澄清** | 要解决什么问题、给谁用、不做什么 | 一句话目标 + 范围边界 |
| **需求** | 用户可见行为、规则、异常 | 需求列表、验收标准（你现在的「需求设计」偏这层） |
| **方案 / 架构** | 技术选型、模块划分、数据怎么存 | 结构图、API 草图、表结构草案 |
| **详细设计** | 关键算法、状态机、错误码、与第三方的约定 | 与实现同级的说明（可写在注释或 wiki） |
| **实现** | 写代码、接好配置 | PR / 本地提交 |
| **自测** | 按验收路径手动或自动测 | `npm run lint` / `typecheck` / 手工清单 |
| **联调 / 预发布** | 真环境、真数据量、权限 | 部署到预览 / Staging |
| **上线** | 监控、回滚预案 | 正式环境 |
| **运营与迭代** | 根据反馈改需求，回到前面某步 | 变更记录 |

**设计（需求 + 方案）之后**，紧接着通常是：**数据模型 / API 契约 → 实现 → 自测 → 部署**。

## 2. 与你当前项目贴合的极简顺序

1. **需求确认**（已有：`docs/详细设计/留言板需求设计-简版.md`、`docs/详细设计/普通测试需求设计.md` 等）  
2. **对齐实现**：`schema` / `src/data` / 路由是否覆盖需求  
3. **补缺开发**：缺页补页，缺 API 补 API  
4. **跑通**：`lint` + `typecheck` + 主路径点一点  
5. **部署**：Cloudflare / D1 绑定与环境变量  

## 3. 文档写到哪里算「够」

- **只给自己和 AI 看**：短需求 + 代码里的类型/常量即契约， often 足够。  
- **要给别人接手**：补「如何本地跑、环境变量、已知坑」到 `docs/实现/handoff.md` 同类文档。
